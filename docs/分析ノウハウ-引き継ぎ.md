# 地方競馬分析ノウハウ 引き継ぎ（→ 中央競馬/JRAへの適応）

地方競馬で確立した分析手法・ツール・検証規律をまとめる。**手法は流用可能だが、各場の数値・傾向は地方競馬のもの**。JRAはデータ源・バイアスが異なるため、必ずJRAデータで再検証すること。詳細な場別知見は `docs/地方競馬-memory-参考/` を参照。

---
## 1. 分析の全体像（何を当てに行くか）
- ゴール: **軸馬の勝率/複勝率を較正**し、券種ごとに**回収率**で妙味を検証する。
- 大原則(地方で繰り返し確認): **市場は概ね効率的**。勝率は当てられても**単勝は織込済(=軸向き、回収は控除率付近に収束)**。妙味は特定状況に限定的にしか出ない。
- したがって「勝率を当てる」だけでは勝てない。**過小評価が残る局面**（休養明け・特定コネクション・距離変化・馬場状態 等）を回収率で探す。

## 2. 確立した手法（予測ファクター）
- **脚質×コースバイアス**: `逃げIV = 逃げ脚質の勝率 / 全体平均勝率`。地方は前残りが極端(逃げIV 2.9〜4.5)。**四角(コーナー)通過順が勝敗をほぼ決定**。枠順は脚質より影響小。まくりは長め距離(1400m+)で有効。
- **脚質判定は四角ベース**（早めの位置取り＝一/二/三/四コーナーの先頭からの相対位置）。前走脚質が「今走逃げ」の最大予測子。
- **独自レーティング(h2h)**: イロ(Elo)は弱い。**共通対戦相手の着差連鎖(head-to-head, 着差をタイム正規化して中央値)**が強い軸予測。`race-h2h` / `race-axis`。
- **能力z**: レース内でタイム/上がりを標準化(zスコア)。持ち時計・上がり3F。馬場差補正(日次δ)で同一条件の勝ち時計SDが縮小。
- **軸有力度の合成**: h2h(0.5)/脚質(0.2)/騎手(0.2)/枠(0.1) の重み付け和。場により最適重みは変わる。
- **危険ローテ減点**: 例「前走大敗×短間隔続戦」等を減点して軸から外す。
- **コンピ指数ブレンド(地方専用)**: 日刊スポーツ極ウマのコンピ指数を 標準スコア+コンピ0.5 でブレンド。3連複の軸選定で有効だった。**JRAでは別の指数/オッズ・人気を使う**。

## 3. バックテストの規律（最重要・ここで多くが否定される）
- **IS/OOS分離**: 学習期間と検証期間を必ず分ける(`-Split` / `-CalTo`)。インサンプルの好成績は信用しない。
- **過学習警戒**: 「サブ条件を足すと回収率が跳ねる」のはほぼ過学習。**標本数を確認**し、条件を増やすほど疑う。
- **回収率は控除率と比較**: 地方単勝の控除率≈25%(=回収75%が基準)。JRAは券種で約20〜30%。**100%超でも95%信頼区間が100%をまたぐ**なら確変ではない。
- **年度ブレが巨大**: 単年の好成績は偶然が多い。**複数年で頑健か**を必ず確認(地方では1年でひっくり返る例多数)。
- **標本の薄い場別/条件別の高回収はノイズ**。`log`等で「何件か」を必ず出す。
- 確立した結論の型: 「**勝率は当たるが単勝は織込済→軸向き**」「妙味は休養明け・特定厩舎・距離短縮(降級)・馬場悪化での前残り強化 等に限定」。

## 4. ツール（tools/ ・手法は流用可能）
- **汎用(全場 -Venue 指定)**: `keiba-features` / `keiba-card` / `keiba-extract` / `keiba-score` / `keiba-backtest` / `keiba-nagashi` — 軸スコア+能力z(レース内標準化)。既定重み等は地方チューニング。
- **独自レーティング**: `race-h2h`(共通対戦相手h2h) / `race-axis`(軸有力度=h2h+脚質+騎手+枠) / `*-backtest`。
- **出馬表→買い目**: `today-picks.ps1`(危険ローテ除外つき軸+上位相手を買い目CSV出力)。先行可能性アップグレード/前残り場の逃げIV自動判定/診断 `-DebugRno` あり。
- **各場専用(参考実装)**: `kochi-*`(高知) `oi-*`(大井) `sonoda-*`(園田) `monbetsu-*`(門別) `kawasaki-*`(川崎) — 場ごとのバイアス/コネクション/馬場差の分析例。**JRAでは場が違うので作り直し**だが、分析の型は参考になる。
- DB接続パターン: 各 .ps1 冒頭で `共通/appsettings.json` の DefaultConnection を読む(`Get-Content ... | ConvertFrom-Json`)。BOM付きUTF-8で保存。

## 5. データ基盤（共通/EF Core）
- `共通/Data/DBContext.cs` が `appsettings.json` の DefaultConnection を読む。DbSet: `競走結果`(着順/馬番/タイム/コーナー通過順…) `レース情報`(出馬表/距離/騎手/馬体重…) `払戻金`(馬券/組番/金額) `当日メニュー`(発走時刻) `リアルタイムオッズ`(単勝オッズ/人気) `馬情報` ほか。
- 取込は `ConsoleApp`(CLIサブコマンド: fetch-*, fetch-range 等)＋`月別開催日程`(スクレイパ)。**いずれも地方サイト向け＝JRAは新規実装が必要**。
- 注: リアルタイムオッズは当日分のみ保持(履歴が要るなら退避テーブルを設計)。

## 6. JRA(中央競馬)への適応の指針 ⚠️
地方の数値をそのまま当てはめないこと。JRAの主な違い:
- **場=10場**(札幌/函館/福島/新潟/東京/中山/中京/京都/阪神/小倉)、**芝・ダート両用**、グレード制、ハンデ・別定。
- **前残りは地方より弱い**(直線が長い東京/新潟外回り等)。逃げIVは小さく、差し・追込も決まる。コース・回り(左右)・内外回りで傾向が大きく変わる。
- **フィールドが大きく市場が厚い→効率的**な可能性が高い。妙味探しは一層シビアに。
- **データ源**: JRA-VAN(有料・正確) / netkeiba 等。リアルタイムオッズも別経路(JRAはIPAT)。**取込基盤(ConsoleApp/月別開催日程に相当)をJRA用に作る**のが最初の仕事。
- **投票自動化**: 楽天競馬(RakutenVote)は地方専用。JRAはIPAT/即PAT。当面は分析に集中し、投票自動化は別途。
- まず: ① `中央競馬` DB作成 → ② JRAの 競走結果/レース情報/払戻金 取込 → ③ `keiba-*`/`race-*` の汎用ツールを `-Venue`(JRA場)で回し、脚質×コース・h2h・能力z をJRAで再較正 → ④ バックテストで回収率検証(規律は §3)。

## 7. 環境の罠（再掲・JRAでも同じ）
- .ps1 は **UTF-8 BOM付き**(PS5.1のShift-JIS誤読対策)。`??` 不可→`Nz`。`$arr +=` 不可→`List[object].Add`。ハッシュ内条件は `$(if(){}else{})`。
- appsettings.json(saパス)・secrets.local.json は **git追跡外**。テンプレから作る。
- MSBuild/実行中exeのファイルロックでビルド失敗→`taskkill` してから再ビルド。

---
**要点**: 「手法(h2h・脚質×コース・能力z・バックテスト規律)」は資産。「各場の数値・コネクション・前残りの強さ」は地方固有でJRAには当てはまらない。JRAデータを入れて、同じ規律で一から較正・検証すること。
