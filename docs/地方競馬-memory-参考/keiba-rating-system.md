# 地方競馬 独自レーティング

[[chihou-keiba-db]] のDBを使った馬の強さ評価。tools/ に実装(全てBOM付きUTF-8、接続文字列はappsettings.jsonから読む)。

## 方式と検証(2025-09〜2026-06でバックテスト、最上位馬の成績)
- **イロ(elo-rating.ps1)**: 着順ベース総当たり。予測力は弱い(勝率17.0%/複勝46.4%)。時計・斤量・展開を見ないため。場間較正の土台としてのみ有用。
- **共通対戦相手h2h(race-h2h.ps1)**: 同一レース内の着差(=勝ち時計比%で距離正規化、馬場・展開フリー)を、直接対決→同レース共通相手→近走共通相手 で連鎖。**強い予測子**(大井 勝率26.3%/複勝55.4%、高知 29.3%/62.1%)。ただし単勝回収は79〜88%で控除前提どまり=市場効率的。軸選び向き。
- **軸有力度(race-axis.ps1)**: h2h(地力)に 脚質適性(馬の近走脚質×場×距離の脚質別勝率)・騎手(場勝率+継続/乗替)・枠適性(内中外勝率)をZ加重合成。**追加で確度向上**: 大井28.4%/57.3%、高知32.2%/63.6%。

## 較正済みの重み
- axis-backtest.ps1 で重みスイープ。**h2h0.5/脚質0.2/騎手0.2/枠0.1 が大井・高知・佐賀の3場とも最良〜僅差**(race-axisの既定値)。場別に変える必要なし(#2は不要と確認)。
- **相手クラス(近走賞金水準)補正は効果限定的**(#1): 大井で複勝/回収が微改善、高知は効果なし、佐賀は同等。クラスを重くすると全場で悪化。h2hが共通対戦相手連鎖で相手強度を既に内包+クリップ/中央値で外れ抑制済みのため冗長。既定は据え置き。

## 精緻版・馬券妙味の検証(2025-09〜2026-06)
- **相手h2h強度で加重した精緻版h2h(axis-bet-backtest.ps1)は効果なし**: 大井28.5/通常28.4、高知32.1/32.2。h2hの共通対戦相手連鎖が相手強度を既に内包するため。クラス補正も同様に無効(#1系は全て不発)。
- **馬券回収率(軸=軸有力度#1、ワイド相手は軸有力度で自動選定)**: 高知が最良で単勝92%/複勝90%、大井は77〜80%。ワイド軸-#2#3(2点流し)が比較的バランス(大井79%/高知82%)。**全戦略<100%=市場効率的**。妙味を出すには人気薄×高軸の選別が要るがオッズ未保存で不可。
- 実用結論: 軸有力度は**強力な軸選定器(高知単勝の的中32%/複勝62%)だが単独で利益化はしない**。オッズ取得が次の鍵。

## 馬体重(2走増減)の妙味
- 「今走馬体重 − 前々走馬体重 ≤ -10kg」を検証(単勝、2025・2026の4-6月)。当初の「連続減=除外/減点」は**逆効果**(園田で減点するほど回収率悪化120→84%)。
- 実際は **-10kg以下は過小評価の妙味**: 全場平均でも≤-10が回収率最高(75〜90%)・勝率高め。**高知が突出**(≤-10で勝率約2倍15〜20%、単勝回収率 両年100%超=141%/119%)。佐賀・川崎も弱い正。**大井・金沢・浦和・盛岡は逆効果**(≤-10の方が劣る)。園田・名古屋・水沢等は年で逆=ノイズ。
- 実装: 減点せず **positive「妙味」フラグ**を高知/佐賀/川崎のみ点灯。today-picks(★妙味)/race-axis(妙味列)/race-profile-screen(妙味減列)。馬体重はレース情報、高知は計測なし(0)の日があり点灯しない事あり。trio-backtest に -WeightPenalty(検証用、減点は逆効果)。

## 調教師ローテ(trainer-rotation.ps1)→ 軸有力度に統合
- tools/trainer-rotation.ps1: 調教師×(出走間隔/距離変化/乗替/馬体重/前走着順)を単要因+C(5,2)=10通りの2要因で集計、勝率/複勝/単回収。必勝/勝負気配なしを別表。-PairMinN で2要因セルの最低標本(既定10)。
- **最重要・両場再現の頑健ルール**: 「**前走6着以下 × 中3週以内(≤27日)の続戦**」は好厩舎でも勝率が平均の25〜40%に沈む(高知:宮川浩/宮川真/田中守、園田:諏訪貴 n235/橋本明/高馬元 等、大標本)。逆に**大敗→休明け・大幅間隔開け**は減点しない(保利幸 休明け・坂本和 中4-6週/前6着は必勝側)。
- 必勝側は場で主役が違う: 高知=細川忠「減量×距離変化」/宮川真「中3週」、園田=保利幸「休明け」。n10〜15の超高回収(1300%超)は過学習で軸根拠にしない。
- **前6着×続戦の減点ルールは高知/園田/大井/佐賀/川崎の5場すべてで頑健**(普遍): 大井 橋本馬 同距離/前6着 n243 勝率2.5%/平均8.4%、佐賀 中川竜 中2週/前6着 n454 勝率3.5%/平均9.6%(最大標本)、川崎 高月賢 乗替/前6着 n250。いずれも勝率が平均の20〜40%。休明け/中4-6週は逆に必勝側(嶋田幸 休明け 等)。
- **race-axis.ps1 / today-picks.ps1 に減点+▼表示統合**: 前走6着以下かつ間隔≤27日で 軸有力度から -RotePenalty(既定0.2)。休明けは除外。race-axisは前着/間隔/危険列、today-picksは▼ローテ表示。
- **減点幅バックテスト(rote-penalty-backtest.ps1, 2025-09〜2026-06)**: 該当馬は地力(h2h)が低く軸上位に来にくいため**軸が入れ替わるのは1000走中1〜19件と僅少、減点幅0.1〜0.5の感度はほぼゼロ**。収支は5場とも軸への影響小:園田のみ単回収80→89%(+9pt、勝率不変)、高知92→92・大井77→75・佐賀76.5横ばい・川崎77→75.9=他4場は中立〜微減。**本質的価値は軸の自動降格より▼警告表示**(地力上位に見える本命の危険ローテを可視化)。統計的エッジで個別の絶対消しではない(該当馬が勝つ事もある)。
- 副次修正: race-axis の Invoke-Rows が1行返却時にスカラ展開し .Count が空→継続/乗替判定・前走照合が無効化されていたバグを return ,$rows で根治(今後 today-picks 等にも同パターン注意)。

## DB/高速化インフラ(2026-06実施)
- **DB は SQL Express → Developer Edition に移行済**(192.168.168.81\SQLEXPRESS、同名インスタンス・混合認証sa・照合Japanese_CI_AS、接続文字列は不変)。max server memory=50GB、16コア解放。Expressの1.4GB/4コア制限を撤廃。移行手順は tools/migrate-to-developer.sql。バックアップは C:\temp\backup(SQLサービスNT Service\MSSQL$SQLEXPRESSにACL付与済)。
- **性能インデックス3本**: IX_競走結果_馬名(馬名,開催日,レース番号 INCLUDE…)/IX_競走結果_場日R馬番/IX_レース情報_馬名。馬名検索(全ツールのN+1)が46倍、ウォームキャッシュ14倍。
- **ボトルネックはSQL→PowerShellへ移行**: DB層は高速化済、重い処理は h2h全ペア計算等のPS単スレッド演算が壁。
- 高速化の型: ①レース内 馬名→行のbyNameハッシュ索引で Where-Object 線形走査をO(1)化 ②`$arr +=` を List 化 ③近走 Sort は date,key で決定的に(同日複数走の順序がエンジン依存=潜在バグ) ④**5.1 runspaceプールでレース計算を並列**(共有データ参照渡し・読取専用→安全、結果不変)。rote-penalty-backtest は -Parallel 実装済(394→161s, 2.46倍, 結果ビット一致)。
- **pwsh7 は分析スクリプトに不適**: 単スレッドでむしろ遅く(System.Data.SqlClient+legacy cmdlet)、かつ culture/列挙順/浮動小数差で h2h がブレ**結果が変わる**(MinCompare境界でレース数が増減)。並列化は5.1 runspaceで行うこと。

## 実装メモ
- 距離をまたぐ着差は秒のままだと過大評価 → **勝ち時計比%に正規化**し±8%クリップ、共通相手経由は中央値集約(外れ値=弱敵大差対策)。
- 走破時計0(速報のみ)や破損値は除外。コーナー欠落レースは脚質判定不可。
- バックテストは「各レースで要素Zを一度算出→重みを一括スイープ」で高速化。
- tools: elo-rating.ps1 / race-h2h.ps1 / h2h-backtest.ps1 / race-axis.ps1 / axis-backtest.ps1。生成キャッシュ `tools/_*.csv` は.gitignore。
