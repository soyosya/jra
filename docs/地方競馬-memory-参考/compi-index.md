# コンピ指数(日刊スポーツ 極ウマ)取り込み

[[chihou-keiba-db]] に外部データ源としてコンピ指数を追加。日刊スポーツ独自の能力数値(40〜90点、最高90)。レース内で各馬に**指数**と**指数順位**。指数1位の勝率高(中央で指数90のコンピ1位=勝率53%/複勝84%)。地方の順位別的中率/回収率は公開少→**自前DBで集計**が本筋。

## 実装済(2026-06-17)
- **DBテーブル `コンピ指数`**: 開催日/開催場所(場名)/レース番号/馬番/馬名/指数/指数順位 + **取得日時(スナップショット=変遷追跡)** + 頭数(順位正規化) + 取得元。索引: UX(レース×馬×取得日時)/結合用(場日R馬番)/順位用。DDL=tools/compi-schema.sql。着順は既存 競走結果 と (開催場所,開催日,レース番号,馬番) で結合。
- **設定**: `共通/appsettings.json` の `GokuUma`(User/Password/LoginUrl/CompiUrl/Headless/ManualLoginAssistSeconds)。※appsettings.jsonはgit追跡=sa/極ウマパスワードが漏れるので注意(環境変数化推奨)。
- **分析 tools/compi-corr.ps1**: 最新スナップを着順/単勝と結合し、指数順位別・指数値帯別(40-49…90)・変遷別(最初→最新の順位↑↓)の 勝率/複勝率/単回収。空データ時は案内のみ。

## 「買うレース選別」軸の検証(compi-race-filter.ps1, OOS前提)
- **リアルタイムオッズ テーブルは空(0件)=馬ごとオッズ無し→人気乖離は検証不可**。EV代理に Harville期待的中(コンピ順位別勝率を全頭正規化, in-sample較正)を使用。
- 3連単 軸1頭マルチ-相手4頭(36点)を 頭数/軸指数/指数差/期待的中 で層別し IS(〜2026-02-28)/OOS(以降)を並置:
  - **OOSで汎化=実フィルタ**: 頭数7-8(IS73.1/OOS73.4)、Harville期待的中≥0.60(IS75.8/OOS73.8)。**最良=「頭数7-8 & 期待的中≥0.55」: IS75.3/OOS76.3(+9pt, OOS565R)**。
  - **過学習でダメ**: 指数差・軸指数(OOSで崩壊)、相手選定(exact/レンジ分割)、場別(金沢104→72)。
  - **意味づけ**: 地方3連単の払戻率≈75%。全体ALL=67%(コンピ36点はやや非効率)だが、**選別レースは76%で払戻率(75%)を上回る**=平均購入より良い選別に成功(ただし100%超は未達、点数過多が希薄化)。100%超には券種圧縮/オッズ要。tools/compi-race-filter.ps1(-Split でIS/OOS)。
- **規律**: in-sampleの見かけ改善(特に細分化・場別・高分散3連単)は信用せず `-Split`/`-CalTo` でOOSを通ったものだけ採用。汎化したのは「頭数」「期待的中」のみ。
- **点数圧縮+券種(compi-bet-structure.ps1, IS/OOS, 全場)**: 軸=コンピ1位/相手=コンピ上位で3連単/3連複×相手2-4頭を比較。**(1)3連複>>3連単**(的中条件同一・点数1/6で希薄化少→全レースで3連単67→3連複72-78%)。**(2)相手は3頭が最適、4頭目(コンピ5位)は希薄化で逆効果**。**(3)選別レース(頭数≤8&期待的中≥0.55)×3連複相手3頭=IS84.1/OOS77.7、相手2頭(1点)=IS82.4/OOS78.2**。ベース(36点3連単67%)から**OOS+11pt(~78%)へ改善・OOS安定**。地方3連複払戻率≈75%を**わずかに上回る**が100%超は未達(オッズ=人気薄の上乗せが無いと+EV化は困難、リアルタイムオッズ空)。到達点: 券種(3連複)+点数圧縮(相手2-3)+レース選別 で平均超えまで。tools/compi-bet-structure.ps1(-FieldMax/-EhMin/-Split)。

## バックテスト結果(2026-06-18)= コンピは最強の軸シグナル
- **単体シグナル(compi-corr 全場595,778走)**: 指数順位1=勝率41.3%/複勝73.4%、順位2=20.5%/58.4%…単調。指数帯90=勝率69.2%/複勝91.1%、80-89=46.6%/79.3%。**単回収は全帯75〜89%で100%割れ=市場織込済**(妙味でなく軸向き)。compi-corr.ps1 は `$rows`をList化して高速化済(旧:+=でO(n²)ハング)。
- **統合BT(axis-backtest.ps1 `-WithCompi`、5場2025-09〜2026-06、計4,107レース)**: コンピをZ化し6番目の因子に追加(馬番で結合)。**コンピ単独の軸選択が、既存5因子(h2h/脚質/騎手/枠/クラス)モデルを全場で確度上回る**:
  - 本命勝率 標準→コンピ単独: 大井28.4→33.2 / 高知32.2→38.7 / 門別31.5→35.5 / 園田30.7→38.9 / 川崎27.5→35.6(**+4〜8pt**)。複勝も**+6〜9pt**。
  - **最良運用=「標準(.5/.2/.2/.1)+コンピ0.5」**(コンピの確度+h2hの妙味)。単回収は高知94.8が最良、他79〜84で**概ね100%割れ=単勝単体は非妙味**。
  - 結論: **コンピは軸の確度を底上げする起点**(3連系/ワイド/複勝の的中率向上に使う)。単勝で抜くものではない。[[keiba-rating-system]][[keiba-ability-toolkit]] の軸有力度にコンピ項追加が有望(未統合)。
- **券種別BT(compi-bet-backtest.ps1、コンピ順位を軸に機械生成、全場10,557レース2025-09〜2026-06)**: コンピ順位だけで買うと**全券種で控除率に負ける(回収<100%)**。回収率は **複勝本命88%(的中74%) > ワイド/馬連77% > 三連複66-73% > 三連単68-71%**=単純・手広いほど分岐に近く、3連系・点数増ほどROI悪化。的中率重視なら複勝コ1+コ2=91%(回収85)。新ツール: tools/compi-bet-backtest.ps1(-Venue/-From/-To/-MinField/-ByGap)。※新規.ps1はBOM付きUTF-8必須(5.1がANSI誤読)。
- **A=指数差(コ1-コ2)別 `-ByGap`(全場)**: **複勝本命は大差ほど的中・回収↑**(差0-4:65.8%/85.7 → 15-19:80.4%/90.4 → 20+:**85.6%/92.3**)。逆にワイド/馬連は大差だと相手2-4が来ず回収↓(10-14でピーク後低下)。**運用則=大差(差≥15)は複勝/単勝を本命1点で厚く・手広い券種は買わない/接戦は手広く**。
- **相手選定の条件付き複勝率検証(compi-relay-select.ps1、軸指数バンド×相手指数バンド/exact→3着内率、全場)**: **相手の複勝率は相手指数に(標本を増やすと)単調増加**(軸80-84なら相手75-79=74%→…→40-44=4%、exact整数でもほぼ単調=穴スポット無し)。よって「相手複勝率上位4頭」≡「コンピ2-5位」で相手4頭ほぼ一致・成績同一。
  - 追加3方式(従来コンピ2-5位/exact複勝率上位4/指数5ptレンジ分割の各レンジ最良)を3連単36点で比較。**in-sample**ではexact+0.7pt・レンジ分割+3.4pt(回収70.9、金沢104%等)と良く見えるが、**`-CalTo`でアウトオブサンプル(2025-09〜2026-02較正→2026-03以降検証)すると全て消滅**: exact 68.2 vs 従来68.5、レンジ分割 68.5=従来68.5(的中はむしろ低下32%)、金沢104%→72.7%に崩壊。
  - **結論(OOSで確定): 相手選定はコンピ2-5位(指数上位4)が最適。exact指数/レンジ分割は過学習で本物の改善なし。相手の絞りでは妙味は出ない**。残る現実的レバーは場選別/買うレース選別(EV)だが、これらもin-sample場差は変動大→OOS必須。改善余地は相手でなく「買うレースの選別(期待的中/EV閾値)」。表は軸指数×相手指数で各馬3着内率が出るのでレース期待値算出に使える。
- **確率→3連単マルチBT(compi-prob-trifecta.ps1、軸=コンピ1位/相手=コンピ2-5位、3連単 軸1頭マルチ-相手4頭=36点、全場10,339レース)**: コンピ順位→3着内率を較正(順位1=75.9%/2=59.1%/3=45.5%/4=35.4%/5=27.0%、=馬券確率)。成績=**実的中率41.6% / 回収率67.4%**(Harville期待的中52.7%でモデルやや楽観)。**指数差では改善せず(~65-71%でフラット、複勝と違い相手の3着内が必要なため)**。**場依存大=笠松87% > 名古屋73 > 水沢72 …大井61・浦和55**。結論: 36点に広げると控除率(3連単~27.5%)に負ける。妙味化には場選別(笠松級)+相手の絞り/EVフィルタが必要。tools/compi-prob-trifecta.ps1(-Venue/-From/-To/-MinField/-Partners)。
- **C=コンピ×h2h併用 券種別 `axis-backtest.ps1 -BetBacktest`(高知/大井, コンピ単独 vs 標準+コンピ0.5でランキングし軸-相手2-4等を実払戻集計)**: **本命複勝はコンピ単独で十分**(h2h足しても~85-89で不変)。**馬連は h2h併用で一貫改善**(高知78→82.4 / 大井76.2→81.7、相手選びにh2hが効く)。三連複は場依存(高知81.5→**89.2**改善 / 大井は微減)。ワイドは微改善。**=本命=コンピ、相手=h2h併用が筋**。なお全戦略まだ<100%。100%超候補=「大差複勝(92)」×人気薄(オッズ)重ね or 場選別。

## 軸入替/h2h併用 検証(2026-06-18, axis-backtest -AxisSwapBacktest, 6場プール451IS/263OOS)
選別(頭数≤8&コンピ期待的中≥0.55)×3連複相手3頭で軸/相手の選び方を比較(OOS=2026-03以降):
| 方式 | OOS回収 |
|---|---|
| コンピ純(軸=コンピ1位/相手=コンピ2-4) | 70.3% |
| 独自純(h2h+脚質+騎手+枠の軸有力度のみ) | 73.9% |
| **併用blend(標準0.5/.2/.2/.1+コンピ0.5)** | **84.7%** |
| 入替swap(コンピ1位が独自分析で3位より下なら独自1位へハード入替) | 64.8% |
- **結論: h2hは反映すべき。最良は「標準+コンピ0.5」の加重ブレンドで軸も相手も選ぶ=OOS 84.7%(コンピ純70.3に対し+14pt, IS88.5/OOS84.7で両期間一貫)**。地方3連複払戻率75%を大きく超え、これまでで最も+EVに近い。
- **ハード入替(swap)は逆効果(最悪64.8)**=「コンピ1位を独自分析で低評価なら独自1位に置換」という二値入替はダメ。**ブレンドが正解**(h2hが強く反対する時だけ自然に軸が入れ替わる)。ユーザの直感(独自分析でコンピ1位を補強/修正)は正しいが、機構は二値でなく加重。
- **コンピ1位の信頼性**: 単独でも強い(本命複勝はh2h足しても不変)が、3連複の軸/相手ランキングではh2h併用が明確に上。
- **実装済 `tools/compi-today-blend.ps1`**(ブレンド版当日カード): axis-backtest同方式でh2h/脚質/騎手/枠を過去[Date-HistDays(400)..Date-1]から算出、当日エントリー=レース情報(出馬表: 馬番/馬名/騎手/距離)、コンピ=当日最新。**標準+コンピ0.5(h0.5/k0.2/j0.2/d0.1/p0.5)で全頭ランキング→軸=1位/相手=2-4位**、頭数≤8&コンピ期待的中≥0.55で3連複相手3頭(3点)。**h2hがコンピ1位に強く反対する時は軸を自動入替(出力に「コンピN位→h2hで軸入替」表示)**。オッズあれば★妙味。-Date/-CalFrom/-HistDays/-FieldMax/-EhMin/-Venue/-ExportCsv。6/18=7レース、名古屋10R(コンピ2位へ)/門別4R(コンピ3位へ)で軸入替を確認。**※ハッシュ内の条件式は `$(if…)` 必須(`(if…)` はif誤認識でエラー)**。compi-today.ps1(コンピ純)も残置だがblend版が上位。
- **複数日ライブ検証(2026-06-01〜06-16, 各89レース, 両スクリプトに `-Verify`+`VERIFYRAW|version|n|h|stake|ret`(ASCIIで文字化け回避)を出力しBashループ32回で合算)**: **ブレンド回収106.0%(的中49.4%) vs コンピ純69.3%(的中42.7%)**。**ブレンドはこの窓で100%超え**=OOS(84.7 vs 70.3)を実カード経路でも追認。単日(6/17 blend43.3%)はノイズと確定。**結論: 実戦用は compi-today-blend.ps1(標準+コンピ0.5ブレンド・軸入替)で確定**。※89レースは中規模・106%は分散含む→保守的見積りはOOS~85%。ツール完成形: fetch-compi/fetch-odds(自動収集)→compi-today-blend.ps1。VERIFYRAWで任意期間のライブ回収を再検証可。

- **ConsoleApp `fetch-compi`** = `月別開催日程/Services/コンピ指数取得.cs`。プロファイルで chiho_c→未ログインなら待機→compi.phpで対象日のkaicode収集→各 compi_detail を `ParseDetail`(table.umaban解析)→ Microsoft.Data.SqlClient で raw INSERT(取得日時=スナップショット、場名は場コード→場名マスタ正規化)。引数 `--date yyyy-MM-dd / --venue <場名> / --kaicode <code> / --headless / --no-dump`。dumpは既定でDumpDirへ。

- **一括バックフィル `fetch-compi-range`**(取得範囲): 年インデックス `compi.php?year=YYYY` が**その年の全開催日×全場を1ページで持つ**ので年ごとに全kaicode収集→範囲フィルタ→古い順に detail 保存。`--from`(既定2022-01-01)/`--to`(既定今日)/`--venue`/`--dump`(既定OFF)/`--force`/`--sleep`(既定600ms)。**取得済(開催日×場)はスキップ=再開可能**(LoadExistingKeys)。Chromeは300件ごと再起動。2022-10以降=全場、それ以前=南関のみ掲載。推定~5000detail/数時間・無人。
- **既存補完への統合**: `取得範囲` を型付き入口 `取得範囲(DateOnly from, DateOnly to, venue=null, headless=false, dump=false, force=false, sleepMs=600)` にも公開。`fetch-range <from> <to> [full] [compi]` の **`compi` トークン**指定で、keiba.go.jp補完後に同期間のコンピ指数も補完(keiba用Chromeを閉じてから専用プロファイルChromeで実行)。`RunFetchRange` 末尾で `コンピ指数取得.取得範囲(dateFrom,dateTo)` 呼出。nextraceinfo(翌日補完)への組込は未対応(無人ジョブのログイン待ちブロック回避のため要望時に追加)。
- 運用: 初回 `ConsoleApp fetch-compi`(画面で手動ログイン)→以降自動。当日コンピ最終確定は21時頃(出走取消対応)。**未実行=実走テスト**(プロファイル作成+ログイン後の初回取得検証)。一括は `ConsoleApp fetch-compi-range`、補完同時は `ConsoleApp fetch-range <from> <to> compi`。
